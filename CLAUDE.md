# 開発ルール

## プロジェクト概要
- Project Zenith: JEPX電力コスト最適化プラットフォーム（運営: Inbox LLC）
- 本体: app.py（Streamlit単一ファイル）。データ取得: fetch_data.py。通知: send_daily_report.py / send_test_mail.py
- データ: data/spot_YYYY.csv（YYYY=年度。例 spot_2020.csv = 2020/4/1〜2021/3/31）
- スキーマ: date, time_code(1〜48), area(9エリア), price。欠損なし前提だが読込時に検証する

## データ構造の前提（変更厳禁の暗黙知）
- time_code は1始まり30分刻み。8:00=code17、22:00=code44
- 昼夜区分（案A）: 昼間=8:00〜22:00（code17〜44）／夜間=それ以外（code1〜16, 45〜48）
- 年度定義: 4月〜翌3月。1〜3月は前年の年度（fiscal_year = year if month>=4 else year-1）
- 冬季は年跨ぎ（12月＋翌1〜2月）だが年度で括ると同一年度に収まる
- 全CSVをload_dataで結合するため、年度判定はファイル分割位置に依存しない

## UI描画ルール（連動漏れ防止・最重要）
- タブ・グラフを追加/編集したら、必ず次の2フィルタの適用要否を確認する:
  - selected_area（表示エリア）: "全エリア"以外なら df[df['エリア'] == selected_area] で絞る
  - date_range / selected_date（期間）: 期間系の描画は必ず指定範囲で絞る
- 新規描画は既存タブのフィルタ適用パターンを踏襲する（独自実装で連動を欠かさない）
- 集計元データフレームを必ず確認: 単日(day_df)／期間(date_range)／全期間(df)のどれが正しいか
- 絞り込み後は empty チェックし、空なら st.warning で明示（無言の空表示を禁止）
- 画面のタイトル・キャプションに集計条件（エリア・期間・年度）を明記する

## 外部API実装前
- 必ずcurlで疎通確認してから実装
- 確認コマンドをコメントに残す
- Billing/認証状態を事前確認
- 無償枠上限・課金条件・想定月次コストを事前に明示（賄えない場合は代替を同時提示）

## エラー対処
- エラーコードで原因を確定してから修正
- 404: URL/モデル名/API有効化を順番に切り分け
- 429: Billing未設定またはレート制限を確認
- 401: 認証キーの値と渡し方を確認
- 同じ修正を2回試みることを禁止
- 修正前に必ずローカルで再現確認

## 実装原則
- ローカルで動作確認してからGitHub Actionsに投入
- 推測での修正禁止、根拠を確認してから変更
- 変更は最小範囲、1箇所ずつ
- 外部APIはcurl疎通→ローカル実行→Actions投入の順序を守る
- @st.cache_data使用時はデータ更新が最大ttl分遅延する点に留意（再読み込みボタンで即時化）
